## 파일 1: modules/03-quantization.html
title: 모듈 3 · 양자화 — AXIS 학습 세션 / badge: MODULE 03 / h1: 품질을 조금 양보하고 속도와 메모리를 얻는 법
pager: 이전=02-how-llm-works.html(모듈 2 · LLM 동작 원리), 다음=04-serving-engines.html(모듈 4 · 서빙 엔진)

### 섹션 01. 모델의 무게는 어디서 오나
- 모델 = 수십억 개의 숫자(파라미터) 덩어리. 각 숫자를 몇 비트로 저장하느냐가 파일 크기를 정함.
- 기본 정밀도 FP16(16비트) 기준 대략 파라미터 수 × 2바이트: 7B 모델 ≈ 14GB, 30B ≈ 60GB.
- 이대로면 게이밍 노트북(8~12GB VRAM)에 7B도 빠듯 → 줄여야 한다.

### 섹션 02. 양자화 = 정밀한 반올림
- 각 숫자를 더 적은 비트로 표현: 16비트 → 8비트(Q8) → 4비트(Q4). 크기가 절반, 1/4로.
- analogy: 3.14159265를 3.14로 적는 것. 대부분의 계산에선 차이가 안 나지만, 아주 미세한 구분이 필요한 계산에선 오차가 보이기 시작.
- 표기 읽는 법 card: `Q4_K_M` = 4비트 양자화(Q4), K_M은 세부 방식·크기 변형. 실무 기본값으로 가장 널리 쓰는 것이 Q4_K_M. 품질을 더 지키고 싶으면 Q6_K, Q8_0.
- 효과 두 가지: ① VRAM에 들어간다 ② 파일이 작아지니 토큰 속도도 빨라진다(모듈 1의 대역폭 ÷ 크기).

### 섹션 03. GGUF — 로컬 추론용 표준 포맷
- llama.cpp 계열(Ollama, LM Studio 포함)이 읽는 단일 파일 포맷. 양자화된 모델이 이 포맷으로 배포됨.
- HuggingFace에서 모델을 찾을 때 저장소 이름에 "GGUF"가 붙은 것을 받는 이유.
- 같은 모델이라도 Q2~Q8 여러 파일이 한 저장소에 있고, 파일 크기가 곧 필요한 VRAM의 대략적 기준.

### 섹션 04. 필요 VRAM 산식 — 이 계산이 서빙 계획의 절반이다
- card로 공식 제시: 필요 VRAM ≈ ① 모델 파일 크기(GGUF 그대로) + ② KV 캐시(컨텍스트 길이·동시 사용자에 비례) + ③ 작업 버퍼(대략 1~2GB)
- 예시: 8B 모델 Q4_K_M ≈ 파일 4.9GB → 12GB VRAM 노트북이면 KV·버퍼 포함 여유 있게 구동, 컨텍스트도 넉넉히 잡을 수 있음.
- 반례: 24GB 서버에 파일 22GB짜리 모델을 올리면 KV 캐시 자리가 없어 컨텍스트를 거의 못 잡음 → "파일이 들어간다 ≠ 쓸 수 있다".

### 섹션 05. 직접 계산해보기 (인터랙티브 계산기)
- 인라인 JS로 간단 계산기 구현. UI: card 안에 폼.
  - 입력 1: 파라미터 수 B 단위 숫자 입력(기본값 8)
  - 입력 2: 양자화 select — FP16(16bpw)/Q8_0(8.5)/Q6_K(6.6)/Q4_K_M(4.8)/Q3_K_M(3.9)
  - 입력 3: 컨텍스트 select — 4k/8k/16k/32k/64k 토큰
  - 입력 4: KV 부담 select — 가벼운 모델(0.05GB per 1k tok)/보통(0.1)/무거운 모델(0.2)
  - 입력 5: 내 GPU select — 8GB/12GB/16GB/24GB
  - 계산: 무게 = 파라미터 × bpw / 8 × 1.05, KV = (컨텍스트k) × 계수, 합계 = 무게 + KV + 1.5(버퍼)
  - 출력: "모델 무게 XGB + KV 캐시 YGB + 버퍼 1.5GB = 총 ZGB → [여유 있음(합계 ≤ GPU의 85%) / 빠듯함(≤100%) / 들어가지 않음(>100%)]" 색상: 여유=--accent, 빠듯=#facc15, 불가=--wrong
- 주의문(작게): 이 계산기는 감을 잡기 위한 근사치입니다. 모델 구조마다 KV 계수가 다르므로 실제 배치 전에는 반드시 실측합니다.

### 섹션 06. 품질은 얼마나 떨어지나
- 일반적 경험칙: 요약·분류·추출 같은 태스크에서 Q4_K_M은 원본과 체감 차이가 거의 없음. 미세한 논리·숫자 계산·긴 추론은 Q6/Q8에서 더 안정.
- 결론은 하나: 소문이 아니라 내 태스크의 골든셋으로 직접 비교한다(모듈 6에서 방법을 배움).
- axis-box: 우리 연구회의 8월 목표가 바로 이 산식으로 각자 '우리 서버(24GB)에 어떤 모델·양자화·컨텍스트 조합이 최적인지' 계산해보고, 실측과 대조하는 것이다.

### QUIZ (5문항)
1. q: 7B(70억 파라미터) 모델을 FP16으로 저장하면 대략 몇 GB인가? opts: [약 1.4GB / 약 7GB / 약 14GB / 약 70GB] a=2. why: FP16은 파라미터당 2바이트이므로 7B × 2 ≈ 14GB입니다. 그래서 양자화 없이는 게이밍 노트북에 올리기 어렵습니다.
2. q: Q4_K_M에서 'Q4'가 뜻하는 것은? opts: [4세대 모델 / 파라미터를 약 4비트로 저장하는 양자화 / 4GB 파일 / 코어 4개 사용] a=1. why: 숫자 하나를 16비트 대신 약 4비트로 표현해 크기를 약 1/4로 줄인 것입니다. K_M은 세부 방식 표기입니다.
3. q: 양자화가 속도까지 빨라지게 하는 이유는? opts: [GPU 온도가 내려가서 / 모델 파일이 작아져 토큰마다 읽어야 할 메모리 양이 줄어서 / 토큰 수가 줄어서 / 전기를 덜 써서] a=1. why: 생성 속도 상한 ≈ 대역폭 ÷ 모델 크기(모듈 1). 분모인 모델 크기가 줄면 속도가 올라갑니다.
4. q: "GPU 24GB에 파일 22GB 모델"이 실무에서 곤란한 이유는? opts: [파일이 아예 안 열려서 / KV 캐시와 버퍼 자리가 없어 컨텍스트를 거의 못 잡아서 / 라이선스 위반이라서 / 22가 짝수라서] a=1. why: 필요 VRAM = 모델 무게 + KV 캐시 + 버퍼입니다. 무게만으로 꽉 차면 정작 대화·문서를 담을 공간이 없습니다.
5. q: 양자화로 인한 품질 저하를 판단하는 올바른 방법은? opts: [인터넷 평판을 본다 / 내 태스크의 골든셋으로 원본과 양자화본을 직접 비교한다 / 파일 크기가 작으면 무조건 나쁘다고 본다 / 모델 이름이 멋진 쪽을 고른다] a=1. why: 태스크마다 민감도가 다릅니다. 요약·분류는 Q4로 충분한 경우가 많지만, 결론은 항상 내 데이터로 측정해서 내립니다.

## 파일 2: modules/04-serving-engines.html
title: 모듈 4 · 서빙 엔진 — AXIS 학습 세션 / badge: MODULE 04 / h1: Ollama, llama.cpp, vLLM — 뭘 언제 쓰나
pager: 이전=03-quantization.html(모듈 3 · 양자화), 다음=05-rag-agent.html(모듈 5 · RAG와 에이전트)

### 섹션 01. 서빙이란 무엇인가
- 모델 파일(GGUF)은 그 자체로는 그냥 큰 파일. 이걸 메모리에 올려 채팅창이나 API 요청을 받아 답을 돌려주는 프로그램이 서빙 엔진.
- analogy: 모델 파일 = 악보, 서빙 엔진 = 연주자. 같은 악보라도 연주자에 따라 무대(장비) 요구사항과 동시 관객 수용력이 다르다.
- "OpenAI 호환 API"라는 말: ChatGPT API와 같은 형식의 주소로 요청을 받는다는 뜻 → 도구(n8n 등)들이 그대로 붙는다.

### 섹션 02. llama.cpp — 로컬 추론의 본가
- C++로 만들어진 로컬 추론 엔진의 사실상 표준. GGUF 포맷의 본가.
- 강점: 지원 폭이 가장 넓다 — 소비자 GPU, 구형 GPU, 심지어 CPU만으로도. GPU 여러 장에 모델을 나눠 싣는 것도 가능.
- llama-server 실행 파일 하나로 OpenAI 호환 API 서버가 됨. 도커로도 배포.

### 섹션 03. Ollama — 가장 쉬운 시작
- llama.cpp 엔진을 감싸 명령 두 개로 쓰게 만든 도구: `ollama pull 모델명`, `ollama run 모델명`.
- 모델 저장·전환·API까지 자동. 개인 실습의 표준.
- 한계: 세밀한 서버 설정(동시 슬롯, KV 정밀 제어)은 llama-server 직접 실행보다 제한적.

### 섹션 04. LM Studio — 코드 없이 GUI로
- 검색→다운로드→채팅을 전부 화면 클릭으로. 서버 기능도 버튼 하나.
- 비개발자의 첫 로컬 LLM 경험으로 최적. 내부 핸즈온에서 써본 그 도구.

### 섹션 05. vLLM — 운영급, 단 신형 GPU 전제
- 수십~수백 명 동시 사용자를 위한 고성능 서빙 엔진(연속 배칭 등 고급 기법).
- 중요한 제약: 최신 세대 GPU를 전제로 개발되어 구형 GPU(예: 파스칼 세대)는 지원하지 않음.
- 교훈 card: 하드웨어 세대가 소프트웨어 선택지를 제약한다. "무엇을 쓸 수 있나"는 장비를 보고 정해진다 — 서빙 계획 산정에 하드웨어 조사가 먼저 오는 이유.

### 섹션 06. 선택 가이드
- 표: 상황 | 추천 | 이유 — 혼자 노트북에서 실습 | Ollama 또는 LM Studio | 설치·실행 최단 경로 / 팀이 공유하는 사내 서버(구형 GPU 포함) | llama.cpp의 llama-server | 지원 폭·멀티 GPU·API 제공 / 신형 GPU 다수, 대규모 동시 사용자 | vLLM | 처리량 최적화
- axis-box: 우리 구성이 그대로다 — 공유 서버는 llama-server(도커)로 OpenAI 호환 API를 제공하고, 여러분은 그 API에 접속해 실습한다. 개인 노트북 연습은 Ollama/LM Studio면 충분하다.

### QUIZ (4문항)
1. q: '서빙 엔진'의 역할은? opts: [모델을 학습시킨다 / 모델 파일을 메모리에 올려 채팅·API 요청을 받아 처리한다 / 모델 파일을 압축한다 / 인터넷 속도를 높인다] a=1. why: 모델 파일은 악보, 서빙 엔진은 연주자입니다. 파일을 실제로 '쓸 수 있는 서비스'로 만드는 프로그램입니다.
2. q: Ollama와 llama.cpp의 관계는? opts: [경쟁하는 별개 엔진이다 / Ollama는 llama.cpp 엔진을 감싸 쉽게 만든 도구다 / llama.cpp가 Ollama의 GUI 버전이다 / 아무 관계 없다] a=1. why: Ollama의 내부 엔진이 llama.cpp입니다. 편의성(Ollama)과 세밀한 제어(llama-server 직접 실행)의 차이로 골라 씁니다.
3. q: 구형 GPU 서버에서 vLLM 대신 llama.cpp를 쓰는 이유는? opts: [vLLM이 유료라서 / vLLM은 최신 세대 GPU를 전제로 해 구형 GPU를 지원하지 않아서 / llama.cpp가 무조건 더 빨라서 / vLLM은 한국어를 못해서] a=1. why: 하드웨어 세대가 소프트웨어 선택지를 제약합니다. 서빙 계획에서 장비 조사가 첫 단계인 이유입니다.
4. q: "OpenAI 호환 API"의 실무적 의미는? opts: [OpenAI에 데이터가 전송된다 / ChatGPT API와 같은 형식이라 기존 도구들을 주소만 바꿔 그대로 연결할 수 있다 / OpenAI의 허가가 필요하다 / 요금이 부과된다] a=1. why: 형식(규격)이 같다는 뜻일 뿐 데이터는 우리 서버 안에 머뭅니다. n8n 같은 도구를 주소만 바꿔 붙일 수 있어 표준처럼 쓰입니다.

### 섹션 01. CPU와 GPU — 일꾼의 종류가 다르다
- CPU: 소수(수 개~수십 개)의 매우 똑똑한 일꾼. 복잡한 판단·순차 작업 담당.
- GPU: 수천 개의 단순한 일꾼. 같은 계산을 대량으로 동시에.
- LLM 추론의 실체는 거대한 행렬 곱셈 = 단순 곱셈·덧셈의 대량 반복 → GPU가 압도적으로 유리.
- 비유(analogy): 세금 신고(복잡, 전문가 1명=CPU) vs 봉투 1만 장에 우표 붙이기(단순 대량, 아르바이트 1000명=GPU).
- NPU 짧게: 스마트폰·최신 노트북에 들어가는 AI 전용 보조 칩. 전력 효율이 강점이지만, 현재 로컬 LLM 서빙의 주력은 여전히 GPU. "NPU 탑재"라는 광고 문구와 "LLM이 잘 돈다"는 별개라는 점.

### 섹션 02. RAM과 VRAM — 책상이 두 개다
- RAM: CPU의 책상. VRAM: GPU 전용 책상(그래픽카드에 붙은 메모리).
- 모델이 빠르게 돌려면 모델 전체가 VRAM에 올라가야 함.
- VRAM이 부족하면 남는 부분을 RAM/CPU로 넘기는 오프로딩 발생 → 돌긴 도는데 수 배~수십 배 느려짐. "돌아간다"와 "쓸 만하다"는 다르다.
- keypoints로: VRAM 용량 = 올릴 수 있는 모델 크기의 상한 / 오프로딩 = 최후의 수단 / 그래서 GPU 스펙에서 제일 먼저 볼 것은 VRAM GB.

### 섹션 03. 속도를 정하는 진짜 범인 — 메모리 대역폭
- 놀라운 사실: 토큰(글자 조각) 하나를 만들 때마다 GPU는 모델 전체 무게를 메모리에서 한 번 다 읽는다.
- 따라서 생성 속도의 이론상 상한 ≈ 메모리 대역폭(GB/s) ÷ 모델 크기(GB).
- 예시 card: 대역폭 700GB/s GPU + 14GB 모델 → 이론상 최대 초당 50토큰, 실제로는 그 절반 이하가 보통.
- 시사점: 같은 GPU라면 모델(파일)이 작을수록 빠르다 → 모듈 3(양자화)에서 모델을 줄이는 법을 배움.

### 섹션 04. 장비 읽는 법 — 숫자 세 개만 보면 된다
- GPU 이름/세대, VRAM 용량, 메모리 대역폭. (nvidia-smi 명령으로 VRAM 사용량 확인 가능하다는 것 언급)
- 표(table): 장비 예시 | VRAM | 감각 — 게이밍 노트북 RTX 3070 | 8GB | 4B급 Q4 모델 쾌적 / 게이밍 노트북 RTX 4080 | 12GB | 8B급 Q4 쾌적, 14B급 도전 / 서버용 구형 GPU 2장(예: 12GB×2) | 합 24GB | 20~30B급 양자화 모델 분할 탑재 가능
- GPU 세대가 낮으면 최신 소프트웨어(예: vLLM)가 지원을 끊기도 함 → 모듈 4에서 재등장 예고.
- axis-box: 우리 연구회는 구형이지만 대역폭이 좋은 서버 GPU 2장(합 24GB)을 공유 추론 엔진으로 쓴다. 개인 실습은 각자 노트북에서 4B~8B급으로 충분하다.

### QUIZ (4문항, a=정답 인덱스 0-base)
1. q: LLM 추론에 CPU보다 GPU가 유리한 이유는? opts: [GPU가 더 비싸고 최신이라서 / LLM 추론이 단순 계산의 대량 병렬 반복이라 수천 개 코어로 동시에 처리하는 GPU에 맞아서 / GPU에만 인터넷 연결 기능이 있어서 / CPU는 한국어를 처리하지 못해서] a=1. why: 추론의 실체는 거대한 행렬 곱셈입니다. 복잡한 판단(CPU의 강점)이 아니라 단순 계산의 물량전이라 GPU가 유리합니다.
2. q: 12GB VRAM 장비에서 18GB짜리 모델 파일을 실행하면? opts: [실행이 아예 안 되거나, RAM으로 넘쳐서(오프로딩) 크게 느려진다 / VRAM이 자동으로 18GB로 늘어난다 / 아무 차이 없이 잘 돈다 / 모델이 스스로 작아진다] a=0. why: VRAM을 넘는 부분은 CPU/RAM으로 오프로딩되어 수 배~수십 배 느려집니다. '돌아간다'와 '실무에 쓸 만하다'는 다릅니다.
3. q: 토큰 생성 속도의 이론상 상한을 정하는 계산은? opts: [코어 수 × 클럭 속도 / 메모리 대역폭 ÷ 모델 크기 / VRAM 용량 × 2 / 전력 소비량 ÷ 온도] a=1. why: 토큰 하나마다 모델 전체를 메모리에서 읽어야 하므로, 대역폭(GB/s)을 모델 크기(GB)로 나눈 값이 이론상 초당 토큰 수의 상한입니다.
4. q: GPU 스펙에서 "어떤 모델까지 올릴 수 있나"를 정하는 가장 중요한 숫자는? opts: [쿠다 코어 수 / 부스트 클럭 / VRAM 용량 / 팬 개수] a=2. why: 모델 전체가 VRAM에 올라가야 제 속도가 나므로, VRAM 용량이 탑재 가능한 모델 크기의 상한선입니다.

## 파일 2: modules/02-how-llm-works.html
title: 모듈 2 · LLM 동작 원리 — AXIS 학습 세션 / badge: MODULE 02 / h1: 왜 길게 넣으면 느려지고, 같이 쓰면 더 느려질까
pager: 이전=01-hardware.html(모듈 1 · 하드웨어 기초), 다음=03-quantization.html(모듈 3 · 양자화)

### 섹션 01. 토큰 — 모델의 글자 단위
- 모델은 글자가 아니라 토큰(글자 조각) 단위로 읽고 씀. "안녕하세요"가 2~4개 토큰이 되는 식(모델마다 다름).
- 한국어는 영어보다 같은 내용에 토큰이 더 드는 경향 → 같은 문서라도 한국어가 더 '길게' 느껴진다.
- 요금·속도·컨텍스트 한도가 전부 토큰 단위로 계산된다는 점.

### 섹션 02. LLM의 정체 — 다음 토큰 예측기
- LLM이 하는 일은 단 하나: "지금까지 글 다음에 올 토큰 하나"를 확률로 예측. 이걸 수백~수천 번 반복해 문장이 됨.
- 토큰 하나 만들 때마다 모델 전체를 한 번 통과 → 긴 답변 = 그만큼 여러 번 반복 = 오래 걸림 (모듈 1의 대역폭 얘기와 연결).
- analogy: 한 글자 쓸 때마다 사전 전체를 처음부터 훑는 필경사.

### 섹션 03. 컨텍스트 윈도우 — 모델의 작업 기억
- 모델이 한 번에 기억하며 볼 수 있는 토큰 수의 상한(예: 8천, 3만 2천, 13만).
- 초과하면: 오류가 나거나, 앞부분을 잘라 잊어버림.
- 실무 질문으로 연결: "이 규정집 전체를 넣고 질문할 수 있나?" = 규정집 토큰 수 vs 컨텍스트 윈도우.
- 단, 컨텍스트를 크게 잡으면 메모리(KV 캐시)를 그만큼 먹는다 → 아래 05.

### 섹션 04. 입력 읽기와 출력 쓰기 — Prefill과 Decode
- Prefill: 입력 전체를 한꺼번에 읽어들이는 단계. 병렬 처리라 상대적으로 빠르지만 입력이 길수록 오래 걸림 → 긴 문서를 넣었을 때 "첫 글자가 나오기까지의 침묵"이 prefill.
- Decode: 한 토큰씩 순차 생성. 여기가 대역폭 병목.
- 체감 공식 card: 반응 시작까지의 시간 ≈ prefill(입력 길이 비례), 이후 술술 나오는 속도 ≈ decode(초당 토큰).

### 섹션 05. KV 캐시 — 다시 안 읽으려고 쓰는 메모장
- 이미 읽은 토큰들의 계산 결과를 GPU 메모리에 저장해두는 것. 없으면 토큰 하나 만들 때마다 대화 전체를 처음부터 다시 계산해야 함.
- 대가: 컨텍스트가 길수록, 동시 사용자가 많을수록 KV 캐시가 VRAM을 차지.
- 그래서 "동시에 여러 명이 쓰면 느려지는" 이유: ① KV 캐시 경쟁으로 쓸 수 있는 컨텍스트가 줄고 ② GPU가 여러 요청을 번갈아 처리.
- axis-box: 우리 공유 서버도 마찬가지다. 동시 사용 인원과 1인당 컨텍스트 길이를 어떻게 배분할지가 8월에 함께 계산해볼 '서빙 계획 산정'의 핵심이다.

### 섹션 06. 생각하는 모델 — 품질과 속도의 교환
- 최근 모델 중에는 답하기 전에 속으로 긴 '생각(추론 토큰)'을 생성하는 thinking 모드가 있음. 어려운 문제 품질↑, 대신 생성 토큰이 수십 배 → 그만큼 느리고 메모리도 씀.
- 실측 사례 card: 연구회가 내부 사전 실증에서 민원 100건 분류를 돌려봤을 때, 일반 소형 모델은 건당 약 3초, thinking 소형 모델은 건당 평균 59초(최악 65분)였다. 품질이 필요한 곳에만 골라 쓰는 것이 요령.

### QUIZ (5문항)
1. q: LLM이 근본적으로 하는 일은? opts: [정답을 데이터베이스에서 검색한다 / 다음에 올 토큰 하나를 확률로 예측하는 일을 반복한다 / 문법 규칙을 적용해 문장을 조립한다 / 인터넷을 실시간 검색한다] a=1. why: LLM은 '다음 토큰 예측기'입니다. 이 단순한 동작의 반복이 문장이 되고, 그래서 그럴듯한 거짓(환각)도 가능합니다.
2. q: 긴 문서를 넣었을 때 "첫 글자가 나오기까지 한참 조용한" 이유는? opts: [모델이 부끄러워해서 / 입력 전체를 읽어들이는 prefill 단계가 입력 길이에 비례해 걸려서 / 인터넷이 느려서 / 출력 토큰이 소진되어서] a=1. why: 반응 시작까지의 시간은 주로 prefill(입력 읽기)이고, 이후 술술 나오는 속도는 decode(토큰 생성)가 정합니다.
3. q: KV 캐시의 역할은? opts: [모델 파일을 압축한다 / 이미 읽은 토큰의 계산 결과를 저장해 재계산을 피한다 / 오답을 걸러낸다 / 전기 사용량을 줄인다] a=1. why: KV 캐시가 없으면 토큰 하나마다 대화 전체를 다시 계산해야 합니다. 대신 컨텍스트가 길수록, 동시 사용자가 많을수록 VRAM을 그만큼 차지합니다.
4. q: 동시 사용자가 늘면 느려지는 이유로 가장 알맞은 것은? opts: [모델이 피곤해져서 / KV 캐시 등 GPU 메모리를 나눠 쓰고 요청을 번갈아 처리해야 해서 / 토큰 가격이 올라서 / 모델 파일이 커져서] a=1. why: GPU 메모리와 연산을 여러 요청이 나눠 쓰기 때문입니다. 그래서 서빙 계획에서는 '동시 몇 명 × 1인당 컨텍스트 얼마'를 미리 계산합니다.
5. q: thinking(추론) 모드 모델의 특성은? opts: [항상 모든 태스크에서 최선이다 / 품질이 오르는 대신 생성 토큰이 수십 배 늘어 크게 느려질 수 있다 / 메모리를 전혀 쓰지 않는다 / 한국어를 지원하지 않는다] a=1. why: 내부 사전 실증에서도 일반 모델 건당 3초 vs thinking 모델 59초였습니다. 품질이 꼭 필요한 단계에만 선택적으로 씁니다.

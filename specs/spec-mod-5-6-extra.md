## 파일 1: modules/05-rag-agent.html
title: 모듈 5 · RAG와 에이전트 — AXIS 학습 세션 / badge: MODULE 05 / h1: 챗봇을 넘어, 업무를 실행시키기

### 섹션 01. 결정 순서 — 프롬프트 → RAG → 파인튜닝
- 모델이 원하는 대로 답하지 않을 때 손대는 순서. ① 프롬프트를 고친다(비용 0, 대부분 여기서 해결) ② 그래도 안 되면 RAG(지식이 부족한 경우) ③ 그래도 안 되면 파인튜닝(말투·형식 고정, 좁은 태스크 특화 — 마지막 수단).
- 파인튜닝을 마지막에 두는 이유: 비용·시간이 크고, 지식 주입 목적으로는 RAG보다 비효율.

### 섹션 02. RAG — 모델에게 오픈북 시험을 보게 하기
- 문제: 모델은 우리 내부 규정·문서·DB를 모른다(학습에 없었으니까).
- 해법: 질문이 들어오면 관련 문서 조각을 검색해 프롬프트에 붙여서 "이 근거로 답해"라고 시키는 것 = RAG(Retrieval-Augmented Generation, 검색 증강 생성).
- analogy: 암기 시험(모델 단독) vs 오픈북 시험(RAG). 오픈북이면 책에 없는 말을 지어낼 이유가 줄고, 출처도 댈 수 있다.
- 파이프라인 keypoints: 문서를 조각으로 나누고(분할) → 각 조각을 의미 좌표로 변환(임베딩) → 좌표 저장소(벡터 DB)에 넣고 → 질문과 가까운 조각을 찾아(검색) → 프롬프트에 붙인다.

### 섹션 03. 임베딩 — 의미를 좌표로
- 문장을 숫자 벡터(좌표)로 바꾸는 별도의 작은 모델. "비슷한 의미 = 가까운 좌표"가 되도록 학습됨.
- 그래서 단어가 달라도 의미가 비슷하면 검색된다("환불 규정" ↔ "결제 취소 절차").
- RAG 품질의 절반은 임베딩 모델과 문서 분할 방식에서 결정.

### 섹션 04. 챗봇, 워크플로우, 에이전트 — 세 가지 형태
- 표: 형태 | LLM의 역할 | 예시 — 챗봇 | 사람이 묻고 모델이 답한다 | 사내 규정 Q&A / 워크플로우 | 정해진 순서의 한 부품으로 실행된다 | n8n에서 '민원 수신→요약→분류→저장' 자동 흐름 / 에이전트 | 도구를 골라 쓰며 목표까지 스스로 반복한다 | "이 오류 원인을 찾아 보고서 초안까지" 하나로 지시
- 원칙: 워크플로우로 충분하면 에이전트를 쓰지 않는다. 예측 가능성은 워크플로우가 높고, 유연성은 에이전트가 높다.

### 섹션 05. 도구 호출(Tool Calling) — 모델에게 손발 달아주기
- 모델이 텍스트 답 대신 "이 함수를 이 인자로 실행해줘"라고 요청하는 능력.
- 이것으로 DB 조회, 사내 검색, 계산, 파일 작성 같은 실제 행동이 가능해짐 — 에이전트의 기술적 토대.
- 로컬 sLM 중에도 도구 호출을 지원하는 모델이 늘고 있고, 모델 선정 시 확인 항목.

### 섹션 06. 루프와 게이트 — 완료를 의심하라
- 에이전트/워크플로우의 신뢰성은 검증 게이트에서 나온다: 모델이 "완료"라고 주장 → 기계적 검사(정규식, 테스트, 규칙)로 증거 확인 → 실패면 자동 재실행.
- 게이트에 LLM이 아니라 정규식·규칙을 우선 쓰는 이유: 결정적(같은 입력=같은 판정)이어서 의심의 여지가 없다.
- 내부 시연 회고 card: PII(개인정보) 비식별화 데모 — 같은 모델인데 '그냥 믿기'에선 전화번호·IP가 남았고, '게이트+재실행 구조'에선 잔여 0건. 모델이 아니라 구조가 결과를 바꿨다.
- axis-box: 10월에 우리 공통 태스크 파이프라인에 이 게이트를 직접 끼워 넣는 실습을 한다. 그 전까지 각자 '내 업무에서 기계적으로 검증 가능한 완료 조건이 뭘까'를 생각해두자.

### QUIZ (5문항)
1. q: 모델 출력이 마음에 안 들 때 올바른 개선 순서는? opts: [파인튜닝 → RAG → 프롬프트 / 프롬프트 → RAG → 파인튜닝 / RAG → 파인튜닝 → 프롬프트 / 무조건 더 큰 모델 구매] a=1. why: 비용이 낮고 빠른 것부터. 대부분은 프롬프트와 RAG 단계에서 해결되고, 파인튜닝은 좁은 특화가 필요할 때의 마지막 수단입니다.
2. q: RAG가 해결하는 핵심 문제는? opts: [모델의 속도가 느린 것 / 모델이 우리 내부 문서·지식을 모르는 것 / GPU 메모리 부족 / 토큰 요금] a=1. why: 관련 문서 조각을 검색해 프롬프트에 근거로 붙여줌으로써, 학습에 없던 사내 지식으로 답하게 만듭니다.
3. q: 임베딩 모델이 하는 일은? opts: [문장을 의미 좌표(벡터)로 바꿔 비슷한 의미끼리 가깝게 만든다 / 문장을 번역한다 / 문서를 압축한다 / 오타를 고친다] a=0. why: '비슷한 의미 = 가까운 좌표'로 변환하기에, 단어가 달라도 의미로 검색할 수 있습니다. RAG 검색 품질의 핵심 부품입니다.
4. q: "매일 아침 민원 목록을 요약해 분류하고 시트에 저장"하는 자동화에 가장 알맞은 형태는? opts: [챗봇 / 정해진 순서의 워크플로우 / 완전 자율 에이전트 / 파인튜닝] a=1. why: 순서가 정해진 반복 업무는 워크플로우가 정답입니다. 에이전트는 순서를 미리 정할 수 없는 유연한 문제에 씁니다. 워크플로우로 충분하면 에이전트를 쓰지 않습니다.
5. q: 검증 게이트에 LLM 심판보다 정규식·규칙 검사를 우선하는 이유는? opts: [LLM이 비싸서만 / 같은 입력에 항상 같은 판정을 내는 결정적 검증이라 의심의 여지가 없어서 / 정규식이 더 똑똑해서 / 규정상 의무라서] a=1. why: 게이트의 생명은 확실성입니다. 기계적으로 판정 가능한 것(전화번호 패턴, 테스트 통과 여부)은 결정적 검사로, 그럴 수 없는 품질 판단만 LLM 심판으로 보냅니다.

## 파일 2: modules/06-evaluation.html
title: 모듈 6 · 평가 — AXIS 학습 세션 / badge: MODULE 06 / h1: 어떤 모델이 "좋다"고 말할 근거
pager: 이전=05-rag-agent.html(모듈 5 · RAG와 에이전트), 다음=07-selfcheck.html(자가진단)

### 섹션 01. 인상이 아니라 측정
- "써보니 괜찮던데"는 데모 몇 번의 인상. 도입 근거가 되려면 같은 문제지로 여러 모델을 채점한 표가 필요.
- 골든셋 = 입력과 정답(모범 답)의 쌍 모음. 50~100건이면 의미 있는 비교 시작 가능.
- 골든셋이 있으면 얻는 것: 모델 비교, 양자화 품질 확인(모듈 3), 프롬프트 개선의 전후 비교 — 전부 같은 문제지로.

### 섹션 02. 민감 데이터 없이 평가하기 — 합성 데이터
- 문제: 평가하려면 실데이터가 필요한데 실데이터는 개인정보 때문에 못 쓴다.
- 해법: 실데이터의 분포(유형 비율, 길이, 난이도)를 분석해 그 분포를 따르는 가짜 데이터를 LLM으로 생성 = 합성 데이터.
- 실사례 card: 연구회의 내부 사전 실증에서 전년도 민원 통계의 유형 분포를 뽑고, 그 분포대로 가상 민원 100건을 생성해 골든셋을 만들었다. 개인정보 0건이지만 실제 업무 난이도를 반영.

### 섹션 03. 무엇을 재나 — 네 가지 지표
- 표: 지표 | 뜻 | 이런 태스크에서 중요 — 정답률(Accuracy) | 정답과 일치하는 비율 | 분류·추출 / 근거성(Groundedness) | 주어진 근거에만 기반해 답했나 | RAG·요약 / 지연(Latency) | 요청부터 응답까지 시간 | 대화형, 실시간 보조 / 처리량(Throughput) | 단위 시간당 처리 건수 | 야간 배치, 대량 처리
- 핵심: 태스크마다 주 지표가 다르다. 야간 배치엔 지연이 덜 중요하고, 실시간 보조엔 결정적.

### 섹션 04. 환각 — 그럴듯한 거짓
- 모델은 다음 토큰 예측기(모듈 2)라서 모르는 것도 그럴듯하게 지어낼 수 있음 = 환각(hallucination).
- 대응: ① RAG로 근거를 주고 ② 출력과 근거를 대조 검사하고 ③ 기계 검증 가능한 부분(숫자, 형식)은 게이트로 잡는다.

### 섹션 05. 채점을 누가 하나 — 기계, 사람, 그리고 LLM 심판
- 우선순위: ① 기계 채점(정답 일치, 정규식, 형식 검사 — 결정적) ② LLM-as-judge(더 강한 모델에게 채점 기준표를 주고 채점시킴 — 요약 품질처럼 기계 채점이 어려운 것) ③ 사람 검수(최종 확인, 샘플링).
- LLM 심판 주의: 반출이 허용된 데이터만 외부 강력 모델로 보낸다. 심판 결과도 샘플링해서 사람이 재확인.
- 실측 교훈 card: 같은 민원 분류 실험에서 품질 1위는 thinking 모델이었지만 건당 59초로 실무 불가 판정. "가장 좋은 모델"이 아니라 "업무 요구(품질 기준+시간 기준)를 만족하는 가장 가벼운 모델"을 고르는 것이 평가의 결론이어야 한다.
- axis-box: 9월에 공통 태스크 2개의 골든셋을 함께 만들고, 우리 서버 모델과 후보 모델들을 같은 문제지로 채점해 '모델 선정 사유서'를 쓴다. 이 모듈이 그 예습이다.

### QUIZ (4문항)
1. q: 골든셋이란? opts: [가장 비싼 모델 목록 / 입력과 정답 쌍으로 이뤄진 평가용 문제지 / GPU 사양표 / 금색 케이블] a=1. why: 같은 문제지로 여러 모델·설정을 채점해야 비교가 성립합니다. 50~100건이면 의미 있는 시작이 됩니다.
2. q: 개인정보가 든 실데이터로 평가할 수 없을 때의 대안은? opts: [평가를 포기한다 / 실데이터 분포를 반영한 합성 데이터로 골든셋을 만든다 / 개인정보를 눈감고 쓴다 / 외국 데이터를 쓴다] a=1. why: 유형 비율·난이도 분포를 실데이터에서 뽑고, 그 분포를 따르는 가상 사례를 생성하면 개인정보 0건으로 실전 난이도의 평가가 가능합니다.
3. q: 야간에 대량 문서를 일괄 처리하는 태스크에서 상대적으로 덜 중요한 지표는? opts: [정답률 / 근거성 / 개별 응답의 지연(latency) / 처리량(throughput)] a=2. why: 밤새 돌리는 배치는 한 건의 응답 속도보다 총 처리량과 정확도가 중요합니다. 태스크마다 주 지표가 다릅니다.
4. q: 모델 선정의 올바른 결론 문장은? opts: [벤치마크 1위 모델을 쓴다 / 품질 기준과 시간 기준을 모두 만족하는 가장 가벼운 모델을 쓴다 / 가장 큰 모델을 쓴다 / 가장 최신 모델을 쓴다] a=1. why: 품질 1위여도 건당 59초면 실무 불가일 수 있습니다(내부 사전 실증의 실측). 요구 조건을 정의하고, 그것을 만족하는 최소 자원의 모델을 고르는 것이 평가의 목적입니다.
